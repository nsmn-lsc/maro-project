# Log_maro_2026-04-20

## Contexto

Durante la sesión se revisó el problema de acceso de usuarios validado por CLUES en un servidor distinto al original. El hallazgo principal fue que el despliegue nuevo había recreado la base de datos usando únicamente `database/schema.sql`, pero ese archivo no incluía las tablas `cat_unidades` ni `usuarios`.

El flujo actual de autenticación para unidades no consulta la tabla `usuarios`; valida directamente la CLUES contra `cat_unidades`. Por esa razón, en el servidor nuevo no existía una tabla base contra la cual comparar el identificador capturado en el login.

También se confirmó que el alta de pacientes vuelve a depender de `cat_unidades`, por lo que el problema no solo afectaba el acceso inicial, sino cualquier operación que requiriera validar una CLUES existente en catálogo.

## Decisiones Técnicas

### Reconstrucciones

- Se reconstruyó el diagnóstico del flujo real de acceso en lugar de asumir que la autenticación usaba la tabla `usuarios`.
- Se rastreó el recorrido completo desde la pantalla inicial hasta el endpoint que valida la CLUES.
- Se confirmó que la tabla faltante en el servidor nuevo no provenía de una migración dentro de `database/`, sino de un script de seed separado.
- Se definió que, para un despliegue desde cero, el sistema necesita dos fases distintas:
  - creación de estructura de base de datos;
  - carga de catálogo y generación de usuarios.

### Refactorizaciones

- Se refactorizó `database/schema.sql` para incluir explícitamente las tablas `cat_unidades` y `usuarios` como parte del esquema base.
- Se refactorizó `package.json` para exponer un comando operativo único que permita poblar catálogo y usuarios sin invocar manualmente el script TypeScript.
- Se dejó alineado el proyecto para que un entorno nuevo no dependa de conocimiento implícito sobre scripts no documentados.

## Código/Configuraciones clave

- Flujo de acceso por CLUES en la pantalla inicial:
  - `src/app/inicial/page.tsx`
  - Si el usuario no entra por rutas estatal o regional, el valor capturado se envía a `/api/unidades/[clues]`.

- Validación real de CLUES:
  - `src/app/api/unidades/[clues]/route.ts`
  - Consulta ejecutada: `SELECT CLUES, UNIDAD, REGION, MUNICIPIO, NIVEL FROM cat_unidades WHERE CLUES = ? LIMIT 1`

- Validación adicional durante el alta de pacientes:
  - `src/app/api/pacientes/route.ts`
  - Consulta ejecutada: `SELECT clues, unidad, region, municipio FROM cat_unidades WHERE clues = ? LIMIT 1`

- Script que originalmente creaba y poblaba tablas fuera del esquema principal:
  - `scripts/seed-unidades-usuarios.ts`
  - Responsabilidades:
    - crear `cat_unidades`;
    - crear `usuarios`;
    - cargar datos desde `unidades_regiones.csv`;
    - generar usuarios por CLUES, región, estado y admin.

- Cambio realizado en el esquema principal:
  - `database/schema.sql`
  - Se agregaron:
    - tabla `cat_unidades`;
    - tabla `usuarios`;
    - índices básicos para búsqueda por región, municipio, nivel, nivel/región y nivel/CLUES.

- Cambio realizado en scripts npm:
  - `package.json`
  - Se agregó el script:
    - `db:seed-unidades-usuarios`: `tsx scripts/seed-unidades-usuarios.ts`

- Secuencia operativa resultante para un servidor nuevo:

```bash
npm install
npm run db:setup
npm run db:seed-unidades-usuarios
```

- Interpretación técnica del nuevo flujo de despliegue:
  - `db:setup` crea la estructura completa de tablas necesarias.
  - `db:seed-unidades-usuarios` carga el catálogo de CLUES y genera registros de usuarios.
  - Con solo `db:setup`, las tablas existen pero `cat_unidades` queda vacía.
  - Con ambos pasos, la validación de acceso por CLUES queda operativa.

## Cambios implementados — segunda parte (mismo día)

### Autenticación real con bcrypt

Se realizó la migración completa del sistema de autenticación para usar la tabla `usuarios` con contraseñas bcrypt.

**`bcrypt` instalado** como dependencia de producción; `@types/bcrypt` como dependencia de desarrollo.

**`scripts/seed-unidades-usuarios.ts`** — reescrito:
- Sustituye `createHash("sha256")` por `bcrypt.hash(password, 12)`.
- Genera contraseñas aleatorias de 20 caracteres con `crypto.randomBytes`, sin patrón predecible.
- Todos los registros de `usuarios` se insertan con `activo = true` y `must_change_password = true`.
- Al finalizar escribe `credenciales-bootstrap.csv` en la raíz del proyecto con las contraseñas temporales en texto claro, para entrega segura al administrador.
- `credenciales-bootstrap.csv` fue agregado a `.gitignore`.
- La función `ensureTables()` se conserva como fallback (sigue creando las tablas si no existen), pero ahora las tablas ya viven en `database/schema.sql`.

**`src/app/api/auth/login/route.ts`** — endpoint unificado nuevo:
- Acepta `POST { usuario, password }`.
- Consulta `usuarios` por `username`, verifica `activo`, compara con `bcrypt.compare`.
- Para nivel CLUES hace JOIN con `cat_unidades` para obtener datos de la unidad.
- Actualiza `last_login_at = NOW()` de forma asíncrona.
- Devuelve sesión normalizada con `userId`, `rol`, `mustChangePassword` y datos de unidad.
- Aplica tiempo constante ante usuario inexistente para evitar enumeración.

**`src/app/inicial/page.tsx`** — refactorizado:
- La lógica de tres rutas (estatal → regional → CLUES directo) fue reemplazada por una sola llamada a `POST /api/auth/login`.
- Si `mustChangePassword === true`, redirige a `/cambiar-password`.
- Redirección final según `rol`: `estatal → /estatal`, `regional → /region`, CLUES por nivel numérico.

### Cambio de contraseña en primer acceso

**`src/app/api/auth/password/route.ts`** — endpoint nuevo:
- `PATCH { userId, currentPassword, newPassword }`.
- Verifica contraseña actual, valida mínimo 10 caracteres y que la nueva sea distinta.
- Guarda nuevo hash bcrypt y pone `must_change_password = false`.

**`src/app/cambiar-password/page.tsx`** — página nueva:
- Diseño de dos columnas consistente con `/inicial`.
- Campos: contraseña temporal (actual), nueva contraseña, confirmación.
- Indicador visual de fortaleza con 5 criterios (longitud, mayúscula, minúscula, número, símbolo).
- Botones de mostrar/ocultar por campo.
- Validación inline de coincidencia en tiempo real.
- Al guardar, actualiza `mustChangePassword` en `localStorage` y redirige al módulo según rol.
- Si no hay `userId` en sesión, redirige a `/inicial` al montar.

## Pendientes

- **Deprecar `/api/auth/estatal` y `/api/auth/regional`** — los endpoints viejos ya no son invocados desde el frontend, pero siguen existiendo. Pueden eliminarse una vez confirmado que no hay otros clientes que los consuman.
- **Correr el seed en producción** — `npm run db:seed-unidades-usuarios` debe ejecutarse en el servidor con la base de datos real para poblar `usuarios` con hashes bcrypt. Las contraseñas SHA-256 del seed anterior quedan inválidas con el nuevo sistema de login.
- **Entregar `credenciales-bootstrap.csv`** por canal seguro al administrador y eliminar el archivo del servidor tras la entrega.
- **Verificar compatibilidad del campo `password_hash`** — el esquema nuevo define `VARCHAR(255)`; si la base de datos en producción todavía tiene `CHAR(64)` del seed anterior, se debe correr un `ALTER TABLE usuarios MODIFY password_hash VARCHAR(255)` antes del seed.
- **Página `/cambiar-password` — prueba de flujo completo** — validar que el ciclo login → must_change_password → cambio → redirección funciona de extremo a extremo con un usuario real en la base.
- **Evaluar si conviene migrar** la creación de `cat_unidades` y `usuarios` también a archivos de migración versionados dentro de `database/migrations/`.
- Confirmar en el servidor nuevo que `.env.local` apunta a la base correcta antes de correr el seed.
